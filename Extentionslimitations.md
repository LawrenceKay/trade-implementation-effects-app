# Known limitations and next steps

- **Depth weighting — RESOLVED (2026-07-14)**: `centrality_pipeline.py`'s `load_dta_bilateral_depth()` now reads `data/raw/DTA 2.0 - Vertical Content (v2).xlsx` (`STATA` sheet), counting the 1,007 binary provisions coded 1 for the deepest agreement between each country pair (the sheet's 64 categorical/multiple-choice provisions are excluded — they're not presence/absence codings). Joined to the `Bilateral Information` sheet by WBID. This is on a 0–1,007 scale, not Fan et al.'s 0–52 Hofmann et al. (2017) scale or the "0–48" figure previously noted here — DTA 2.0's provision set is more granular than DTA 1.0's. Eigenvector centrality is invariant to a uniform rescaling of edge weights, so relative rankings are unaffected, but absolute depth values are not comparable to Fan et al.'s published figures.
- **Natural centrality all-zero bug — RESOLVED (2026-07-14)**: root cause was `safe_eigenvector_centrality()` only caught `nx.PowerIterationFailedConvergence` to trigger its power-iteration fallback, but disconnected graphs raise `nx.AmbiguousSolution` instead — so the fallback never ran and the generic `except Exception` branch silently returned all zeros. The natural/intra-regional subgraph splits into 8 disconnected components, one per WB region (largest: Europe & Central Asia, 49 nodes) — a single global power iteration would converge to the dominant eigenvector of the largest component only, numerically underflowing every other region to ~0. Fixed by adding `eigenvector_centrality_per_component()`: computes eigenvector centrality independently within each connected component, rescales each component's unit-normalised vector by that component's own dominant eigenvalue (so denser/richer regions still score higher, matching what a global computation would do if it could), then renormalises the combined vector to unit length so `n_centrality` stays on a comparable scale to `overall_centrality`/`nn_centrality`. Verified: 0 negative values, 0 NaNs; the only remaining zeros (11 of 203 countries — Faroe Islands, Hong Kong, Taiwan, Andorra, Monaco, etc.) are legitimately isolated single-node components with no intra-regional FTA partner, not underflow artifacts.
- **Base network source mismatch — RESOLVED (2026-07-14)**: Fan et al.'s network (who is connected to whom, not just edge depth) is built directly from World Bank DTA data. Ours **used to be** built from DESTA (`desta_dyads.csv`), with WB DTA layered on only as an optional depth overlay — DESTA and WB DTA catalogue overlapping but non-identical sets of agreements, so network topology could differ from Fan et al.'s before depth-weighting was even considered. Migrated to WB DTA 2.0 as the sole source across the whole app (both `centrality_pipeline.py` and `network_example.py`'s previously-separate, independent DESTA loader). DESTA has zero remaining references anywhere in the codebase, and `data/raw/desta_dyads.csv` has been deleted (2026-07-14) along with the now-unused `pycountry` dependency (only ever used for DESTA's numeric-ISO conversion) and the separate `desta_design` institutional-quality metric in `pipeline.py`/`score.py` (which was already a silent no-op — its source file, `data/raw/desta.csv`, never existed). The sub-bullets below record each step of the migration in order.
  - **Decided (task 1 of the migration): which WB DTA 2.0 rows count as a current network edge.** The app wants a live snapshot (is this agreement in force *now*), not a cumulative "ever existed" network — so filtering is by the `Agreements` sheet's `Status` field, not by historical year presence. `Status == "In Force"` is the base criterion; this correctly **excludes** the 16 agreements marked "Inactive" (e.g. NAFTA, superseded by USMCA in 2020 — USMCA is separately listed "In Force" and will carry the current US–Canada–Mexico relationship instead) and the 1 marked "Early Announcement" (EU–MERCOSUR, WBID 397, never actually ratified — confirmed by all 216 of its `Bilateral Information` rows having `year = NaN`, i.e. no real entry-into-force date). This is **asymmetric** — defined by the *partner's* classification, not requiring the focal country itself to match. (The symmetric option would have been actively broken, not just a worse fit: it would make `ED_Centrality` trivially zero for every developing country by construction, since a developing country can never have an edge where *both* ends are developed — gutting the regression, which runs `ED_Centrality`/`ING_Centrality` across the whole sample, developed and developing countries alike.)   - **Structural nuance surfaced during implementation**: because the classification is asymmetric, the same edge can belong to *different* categories depending on whose perspective you're centered on — a developing country connected to a developed one sees that edge as part of its "developed network," while the developed country sees the identical edge as part of *its* "developing network." Eigenvector centrality needs one fixed shared graph per computation, not a per-node-varying edge set, so this doesn't translate into a clean partition the way natural/non-natural does. Resolved by building two *overlapping* subgraphs rather than a partition: `G_ED` = every edge touching at least one Developed country; `G_ING` = every edge touching at least one Developing country. Mixed (Developed–Developing) edges appear in both graphs, which is the correct computable analogue of the paper's per-partner definition, even though it isn't a strict partition like the region split.

## Knowledge Base

This is infrastructure for the not-yet-built Trade Economist advisory agent (see CLAUDE.md Section 8, "Possible extensions and known limitations"). The scaffolding below exists on disk at `knowledge_base/` but is **not wired into the live app** — `network_example.py` and `centrality_pipeline.py` have zero references to it. Grounding the agent's answers in this literature corpus, with inline source citations rather than unsupported claims, is how that extension would be implemented.

**Dependency note (2026-07-16):** `chromadb` and `sentence-transformers`, which `ingest.py` and `retrieve.py` both need, were removed from the repo's root `requirements.txt` when trimming it for a lean Streamlit Community Cloud deploy of `network_example.py` (which doesn't need them). `ingest.py` also imports `pypdf`, which was never in `requirements.txt` to begin with. Anyone actually running the knowledge-base scripts needs to `pip install chromadb sentence-transformers pypdf` separately — this isn't yet captured in its own requirements file.

**Location.** `knowledge_base/` at the project root.

**Structure.**

```
knowledge_base/
  raw/                        # Source documents before ingestion
    fta_networks/             # Papers on FTA network structure and trade flows
    economic_complexity/      # Papers on ECI, product space, complexity theory
    agreement_depth/          # Papers on PTA depth, provisions, and enforcement
  vectordb/                   # ChromaDB persistent store (gitignored)
  sources.csv                 # Metadata registry for all ingested documents
  ingest.py                   # Ingestion script: chunk → embed → store
  retrieve.py                 # Retrieval helper: query → top-k chunks
```

**Sources.csv schema.** One row per document.

| Column | Description |
|--------|-------------|
| `id` | Unique slug (e.g. `fan_etal_2025`) |
| `title` | Full title |
| `authors` | Comma-separated author surnames |
| `year` | Publication year |
| `journal` | Journal or publisher |
| `theme` | One of: `fta_networks`, `economic_complexity`, `agreement_depth` |
| `layer` | `academic` or `policy` |
| `file` | Relative path to the raw document |
| `licence` | Licence or access status |
| `notes` | Any ingestion caveats |

**Ingestion pipeline (`ingest.py`).** Reads each document in `raw/`, splits into chunks of ~500 tokens with 50-token overlap, embeds using `sentence-transformers/all-MiniLM-L6-v2`, and upserts into a ChromaDB collection named `trade_knowledge`. Metadata stored per chunk: `id`, `authors`, `year`, `title`, `theme`.

**Retrieval helper (`retrieve.py`).** Accepts a query string and optional theme filter; returns the top-k chunks as a formatted string with inline source labels ready to paste into a prompt.

**Seed corpus.** The four papers listed in the README's "Complementary literature" section (Fan et al. 2025; Sopranzetti 2018; Yang & Liu 2024; Guo 2026) are the initial seed documents. All are publicly available. Additional documents should be cleared for licence before ingestion (see CLAUDE.md Rules, Section 6).
